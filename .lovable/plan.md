# Predictive Overrun Insights

A single Insights panel above the requests table that flags BoQ lines projected to exceed their budgeted quantity, with a plain-English "why + what to do" line generated by Lovable AI.

## Why this approach

For a construction ERP interview demo, a black-box LLM predicting overruns is a red flag — PMs won't trust it, and outputs vary run-to-run. The credible pattern is **deterministic math for the signal, AI for the explanation**:

- The math (burn rate vs remaining budget) is auditable and always produces the same answer for the same data. This is what a procurement lead would actually defend in a meeting.
- The LLM narrative turns "BOQ-03.04 at 87% consumed, 14-day burn = 12 m³/day, projected to exceed by 34%" into "C30 concrete usage on Tower B has accelerated over the last two weeks; at the current rate you'll overrun the BoQ by ~34% before month-end. Consider pausing new pours or revising the BoQ." That's the AI-native feel without the AI-native risk.

Kept as simple as possible: one panel, one refresh button, no charts, no new route, no settings.

## What the user sees

```text
│ ⚠ BOQ-03.04 · C30 Concrete — projected 34% over      │
│   Consumed 87% of 450 m³ budget in 6 weeks. At the   │
│   last 14 days' rate you'll exceed budget by ~152 m³ │
│   before the BoQ line closes. Consider pausing new   │
│   pours on Tower B or revising the line.             │
│                                                       │
│ ● BOQ-05.01 · Rebar 12mm — trending, watch            │
│   62% consumed, burn rate up 40% vs prior 14 days.    │
└───────────────────────────────────────────────────────┘
```

Only appears when at least one BoQ line is flagged. Severity dot uses accent (golden orange) for `over` and secondary (light blue) for `watch`, consistent with the current palette rule (accent = alerts/urgency only).

## Detection logic (deterministic, runs client-side)

For each BoQ line, from `requests` where `status ∈ {Approved, Ordered, Received}` (Rejected/Requested excluded — they aren't committed spend):

1. `consumed = Σ qty` for that line
2. `pct = consumed / qtyBudgeted`
3. `recentBurn = Σ qty in last 14 days / 14` (per day)
4. `priorBurn = Σ qty in days 15–28 / 14`
5. `remaining = qtyBudgeted - consumed`
6. `daysToExhaust = remaining / recentBurn` (if recentBurn > 0)
7. `accelerating = recentBurn > priorBurn * 1.25`

Flag as:
- **over** — `pct ≥ 0.8` AND `accelerating` AND `daysToExhaust < 30`
- **watch** — `pct ≥ 0.5` AND `accelerating`
- otherwise not flagged

Ranked by projected overrun % descending. Cap at 5 items.

Because seed data is static, the demo will consistently flag the same 1–2 lines — good for a walkthrough.

## AI narrative (Lovable AI Gateway)

One `createServerFn` (`src/lib/insights.functions.ts`) takes the flagged list (computed on the client, sent as input — small payload, no PII) and calls `google/gemini-3-flash-preview` via the Lovable AI Gateway helper. Uses the AI SDK's `Output.object` for a structured `{ id, headline, body }[]` response so rendering is safe.

Prompt is short and constrains the model: "You are a construction procurement analyst. For each flagged BoQ line below, write one headline (≤10 words) and one body sentence (≤30 words) explaining the risk and one concrete action. Do not invent numbers — only reference the figures provided."

Rate-limit / credit-exhaustion errors from the gateway (429 / 402) are caught and the panel falls back to showing the raw math strings without narrative, plus a small "AI explanations unavailable" note. No hidden failures.

## Files

- `src/lib/insights.ts` — pure functions: `computeInsights(requests, boqLines): FlaggedLine[]`. Unit-testable, no React, no AI.
- `src/lib/insights.functions.ts` — `explainInsights` server function; reads `LOVABLE_API_KEY` inside the handler, calls the gateway via the existing `ai-gateway.server.ts` helper (create it if missing per the `ai-sdk-lovable-gateway` pattern).
- `src/lib/ai-gateway.server.ts` — provider helper (new; standard shape from the knowledge file).
- `src/components/requests/InsightsPanel.tsx` — the panel. Uses `useQuery` keyed on the flagged-line IDs so the AI call only refires when the flagged set changes. Manual `Refresh` button invalidates the query.
- `src/routes/index.tsx` — mount `<InsightsPanel />` above the summary chips. No other changes.

No new dependencies (`ai` and `@ai-sdk/openai-compatible` may need `bun add` if not already present — will check first).

## Out of scope

- Charts / sparklines
- Per-line drill-down page
- Persisting insights across reloads
- Notifications / email
- Predicting individual request outcomes (only BoQ-line-level overrun)
- Configurable thresholds in the UI

## Verification

1. Load `/`, confirm the panel appears above the chips with at least one seeded BoQ line flagged (seed data will be nudged if needed so one line crosses the 80% + accelerating threshold).
